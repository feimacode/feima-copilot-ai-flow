# flow-create-command Specification

## Purpose
Enable natural language flow authoring via @flow create and @flow enhance commands, backed by a reusable Prompt-TSX skill that generates valid, self-documenting flows.

## Requirements

### Requirement: @flow create generates valid flow from natural language
The `@flow` chat participant SHALL support a `create` sub-command. When a user invokes `@flow create "<description>"`, the system SHALL generate a valid `.flow.yaml` file in the workspace. The generated flow SHALL be schema-validated against `schemas/flow.schema.json`. The flow SHALL be immediately runnable with `@flow #file:`.

#### Scenario: User creates a simple pipeline flow
- **WHEN** user invokes `@flow create "a code review with separate security and style lenses"`
- **THEN** the system SHALL generate a valid `.flow.yaml` file with at least 2 roles, a `sharedContext` section, and valid YAML syntax

#### Scenario: Generated flow passes schema validation
- **WHEN** a flow is generated by `@flow create`
- **THEN** the output SHALL pass validation against `schemas/flow.schema.json` without errors

#### Scenario: Generated flow is immediately runnable
- **WHEN** a flow is generated and saved to the workspace
- **THEN** invoking `@flow #file:<generated-flow>.yaml` SHALL execute the flow without modification

### Requirement: @flow create generates sharedContext
Every flow generated by `@flow create` SHALL include a `sharedContext` field following the standardized template: ## What This Does, ## When to Use, ## How It Works, ## Example, ## What You'll Get, ## Customize It. The content SHALL be specific to the generated flow, not generic boilerplate.

#### Scenario: Generated sharedContext explains the flow
- **WHEN** a flow is generated for "code review with security and style lenses"
- **THEN** the `sharedContext` SHALL describe code review concepts, list relevant use cases, explain the multi-role approach, and provide customization hints

### Requirement: @flow enhance extends existing flows
The `@flow` chat participant SHALL support an `enhance` sub-command. When a user invokes `@flow enhance <flow-file> --<instruction>`, the system SHALL modify the existing flow file to incorporate the requested enhancement while preserving existing roles, prompts, and structure.

#### Scenario: User adds tool integration to existing flow
- **WHEN** user invokes `@flow enhance story-estimation.flow.yaml --add-jira-integration`
- **THEN** the system SHALL modify the flow to include Jira-related tools and update relevant role prompts, while preserving all existing roles

#### Scenario: User adds iterative stage to pipeline flow
- **WHEN** user invokes `@flow enhance code-review.flow.yaml --add-iterative-review`
- **THEN** the system SHALL wrap the existing roles in a `stages:` block with appropriate iteration configuration

### Requirement: Flow-authoring skill is reusable
The Prompt-TSX component that powers `@flow create` and `@flow enhance` SHALL be implemented as a standalone, reusable skill. It SHALL be invocable independently of the chat participant for use in other contexts (e.g., programmatic flow generation, validation enhancement).

#### Scenario: Skill is documented for power users
- **WHEN** a developer reads the flow-authoring skill documentation
- **THEN** they SHALL understand how the skill makes decisions about structural patterns, role granularity, and tool assignment
